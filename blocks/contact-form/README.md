# Contact Form Block

## Overview

The Contact Form block renders a mock contact form for authored pages. It builds the full client-side form UI in block code, including the heading, description, four required fields, submit button, and inline status messaging. The current implementation does not submit data to any backend service.

## DA Authoring Structure

The block currently reads a single authored content cell for optional display copy. If no content is authored, it falls back to the default heading, description, and CTA label defined in the block JavaScript.

Expected authored structure:

| contact-form |
| --- |
| `## Send Us a Message`<br>`Fill out the form below and our team will get back to you within 24 hours.`<br>`[Send Message]` |

Notes:
- The heading is read from the first authored heading inside the block.
- The description is read from the first authored paragraph inside the block.
- The button label is read from the first authored link or button inside the block.
- The form fields themselves are not authored yet; they are generated by the block implementation.

## Behavior Patterns

### Page Context Detection
- **Default Rendering**: When the block is present on a page, it replaces the authored placeholder row with the decorated mock form UI.
- **Authored Copy Override**: When a heading, paragraph, or CTA is authored in the DA block cell, those values replace the default heading, description, and button label.
- **Fallback Copy**: When no authored copy is present, the block uses its built-in default messaging and placeholders.

### User Interaction Flows
1. **Initial Render**: The block generates unique field and status IDs, builds the form fragment, and attaches submit and input listeners.
2. **Validation Flow**: When the user submits with any empty or invalid field, native browser validation runs and the block shows an inline error message.
3. **Mock Submit Flow**: When all fields pass validation, the block prevents network submission, displays a mock success message, and resets the form.
4. **Status Reset Flow**: When the user edits the form after any shown status message, the inline status is cleared.

### Error Handling
- **Invalid or Empty Field Values**: Uses native `form.checkValidity()` and `form.reportValidity()` before showing a custom inline error state.
- **Missing Authored Content**: Falls back to default copy when the authored heading, paragraph, or CTA is not present.
- **Fallback Behavior**: If no backend integration exists, the block still provides a complete mock interaction flow entirely on the client.
